# CrushPilot PRD 1\.0\.0

## 产品定位

CrushPilot 是一个面向恋爱聊天场景的 AI 聊天助手。

用户输入对方发来的消息、最近几轮对话，或直接描述当前问题，系统结合：

- 当前聊天上下文

- 场景 Skill

- 外部恋爱沟通知识库

- 大语言模型

生成可直接使用的回复建议。

核心问题只有一个：

> 对方刚刚发了这句话，我应该怎么回？
> 
> 

---

## MVP 目标

第一版只验证三个核心能力：

1. 用户能够通过聊天界面持续与 CrushPilot 对话。

2. 系统能够自动选择合适的 Skill。

3. 系统能够检索相关知识并生成自然、可发送的回复。

第一版不追求完整的 Agent 平台，也不覆盖关系管理、长聊天分析等外围功能。

---

## 目标用户

主要用户：

- 不知道如何回复异性消息的人

- 希望回复更自然、幽默或暧昧的人

- 遇到冷场、邀约和关系推进问题的人

- 希望快速获得具体回复，而不是阅读长篇分析的人

---

## MVP 使用场景

### 4\.1 回复建议

用户输入：

```Plain Text
她说：“刚下班，好累。”
我怎么回？
```

系统输出：

```Plain Text
推荐回复：
“那今天先不折腾你了，早点回去休息。”

备选：
1. “听起来今天被工作榨干了。”
2. “辛苦了，今晚适合早点躺平。”

建议：先接住她的情绪，不要连续追问。
```

### 4\.2 回复改写

用户输入：

```Plain Text
太正式了，改自然一点。
```

系统基于当前会话继续生成，不要求用户重新提交上下文。

### 4\.3 冷场处理

用户输入：

```Plain Text
她最近回复越来越慢，我还要继续找她吗？
```

系统调用冷场 Skill，给出是否回复、如何降低压力以及具体回复。

### 4\.4 发起邀约

用户输入：

```Plain Text
我们聊了一个星期，怎么约她周末出来？
```

系统调用邀约 Skill，结合关系阶段和已有话题生成低压力邀约。

## MVP 功能范围

### 5\.1 聊天助手

必须支持：

- 多轮聊天

- 粘贴单条消息

- 粘贴最近聊天片段

- 描述当前关系和目标

- 基于前文继续追问或修改

- 流式展示模型结果

### 5\.2 Skill 路由

第一版只实现四个 Skill：

```Plain Text
reply-suggestion
reply-rewrite
cold-recovery
date-invitation
```

系统根据用户当前输入自动选择 Skill。

允许用户通过自然语言修改结果，例如：

```Plain Text
更自然一点
短一点
不要提问
更暧昧一点
不要这么油
换一个
```

### 5\.3 外部知识库

知识库提供：

- 场景处理原则

- 高质量回复案例

- 失败案例

- 回复约束

- 不同关系阶段的沟通建议

第一版使用本地 Markdown 或 JSON 文件。

目录建议：

```Plain Text
knowledge/
  reply/
  rewrite/
  cold/
  invitation/
```

第一版不使用向量数据库，采用标签、关键词和简单评分检索。

### 5\.4 上下文管理

使用 LangGraph 管理会话状态。

每个会话通过 `thread_id` 标识。

上下文包括：

- 最近若干轮用户与助手消息

- 用户当前目标

- 当前关系信息

- 最近选择的 Skill

- 最近生成的回复

- 历史摘要

默认保留最近 10 至 20 条消息，超出部分生成摘要，避免无限增长。

---

## 输出要求

系统默认输出：

1. 一条最推荐回复

2. 两条备选回复

3. 一句简短判断

4. 必要时给出风险提醒

标准输出示例：

```JSON
{
  "skill": "reply-suggestion",
  "judgement": "对方是在分享状态，当前适合接住话题，不适合强行推进。",
  "recommended_reply": "刚忙完啊，那今晚可以安心躺平了。",
  "alternatives": [
    "听起来今天挺累的，早点休息。",
    "终于下班了，今天是不是被工作折腾惨了？"
  ],
  "warning": null
}
```

禁止默认生成长篇关系分析报告。

## API 设计

### 创建或继续聊天

```HTTP
POST /api/v1/chat
```

请求：

```JSON
{
  "thread_id": "thread_xxx",
  "message": "她说今天有点累，我怎么回？"
}
```

响应采用 SSE 流式返回。

### 获取会话

```HTTP
GET /api/v1/threads/{thread_id}
```

### 获取会话列表

```HTTP
GET /api/v1/threads
```

### 删除会话

```HTTP
DELETE /api/v1/threads/{thread_id}
```

第一版可以暂不实现用户登录，通过匿名设备 ID 区分用户。

---

## 非功能要求

### 性能

- 首个流式字符目标在 3 秒内返回

- 普通请求整体目标在 10 秒内完成

- 单次只检索少量相关知识

### 稳定性

- 模型输出必须使用 Pydantic 校验

- JSON 解析失败时自动重试一次

- 模型调用失败时返回明确错误

- LangGraph 状态保存失败不能导致服务崩溃

### 隐私

- 服务端默认只保存必要的会话信息

- 不自动读取微信或其他应用

- 不自动替用户发送消息

- 不把用户聊天内容用于公开展示

### 安全边界

禁止生成：

- 威胁

- 骚扰

- 跟踪

- 情绪勒索

- 强迫或持续纠缠

- 虚假身份欺骗

- 未成年人性相关内容

## 验收标准

MVP 完成必须满足：

1. 用户可以创建独立聊天会话。

2. 同一 `thread_id` 下能够连续多轮聊天。

3. 系统可以自动识别四个核心 Skill。

4. 每次生成时能够检索相关知识。

5. 系统返回一条推荐回复和两条备选回复。

6. 用户可以继续要求更短、更自然或换一个。

7. 服务端重启后会话能够恢复。

8. 前端能够通过 SSE 展示流式输出。

9. 后端具有基础单元测试和接口测试。

10. 所有第一版非目标功能不得进入主流程。

---

## 开发原则

后续开发统一遵循：

```Plain Text
聊天助手优先
单一主链路
能少做就不多做
先规则检索，后向量检索
先单 Graph，后复杂 Agent
先跑通闭环，后扩展功能
```

任何新增功能必须回答：

> 这个功能是否直接提升“用户现在应该怎么回复”的效果？
> 
> 

如果答案是否定的，则不进入 MVP。
