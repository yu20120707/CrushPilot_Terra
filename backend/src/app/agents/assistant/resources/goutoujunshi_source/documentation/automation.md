# 自动化与代理边界

## 代理说明

| 项目 | 当前实现 |
| --- | --- |
| 触发 | 用户显式调用`$goutoujunshi`，或宿主根据Skill描述匹配相关请求 |
| 所有者 | 用户控制当前任务；仓库维护者维护公开指令 |
| 自动运行 | 只在Codex任务内响应，不存在后台或定时运行 |
| 可读输入 | 当前对话，以及按需加载的仓库Markdown文档 |
| 必需工具 | 无 |
| 外部副作用 | 无；只生成建议文本 |
| 审批 | 现实发送、删除、联系或求助动作均由用户自行决定 |

## 提示词引导与硬边界

`SKILL.md`包含首次建档、情绪支持、知识路由、建议格式和安全边界。这些属于模型引导，并非程序级强制控制。

真正的硬边界来自宿主环境：模型安全政策、文件系统权限、网络权限、工具参数校验和用户批准。仓库不能声称仅靠提示词彻底阻止越权或有害输出。

## 输出契约

常规回答应尽量包含：

1. 对用户情绪和核心矛盾的准确回应。
2. 已知事实、合理推测和关键未知的区分。
3. 明确的首选建议及2–4个理由。
4. 必要时提供不同策略强度、话术和预期代价。
5. 一件下一步、观察窗口或停止条件。

若信息不足，代理最多优先询问会改变建议的关键问题；首次建档例外，需要收集Skill规定的最小资料。

## 失败处理

- 资料缺失：标注未知，不虚构。
- 资料矛盾：指出矛盾并请求纠正关键项。
- 高风险请求：拒绝危险操作部分，提供安全替代。
- 法律或医疗不确定：说明局限，建议核对地区和咨询合格专业人员。
- 工具不可用：继续提供纯文本支持，不假装已执行外部操作。
