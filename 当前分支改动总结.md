## 3. 聊天与 RAG 工作流

当前聊天图为：

```text
用户消息
  -> 输入风险规则 / 模型意图分析
  -> 安全知识词法检索（含同义扩展）
  -> 无命中时只循环一次本地关键词扩展
  -> 结构化模型生成
  -> 可执行危险建议审核
  -> SSE complete 事件 / 会话持久化
```

### 已移除的旧逻辑

- 不再使用 `route_skill`、`SKILLS`、`reply-suggestion` 等固定 Skill 路由。
- 不再按完成后人工切分 `token` 伪装流式生成。
- 不再从 LangGraph checkpoint 全量扫描并反查会话列表。

### 当前实现的关键行为

1. 模型返回 `scene`、`goal`、`features`、`keywords` 和 `risk_flags`；服务端只接受 `unsafe_request` 这一高风险码，忽略其他模型自定义 flag。
2. `疲惫`、`累`、`加班`、`工作`、`邀约`等关键词会作受控同义扩展，例如“疲惫”可扩展为“累、辛苦、休息”。
3. 正常命中和无命中路径都最多两次模型调用：意图分析一次、最终生成一次；无命中不再额外调用模型改写检索词。
4. 输出审核只检查 `recommended_reply` 和 `alternatives` 中的可执行危险建议，避免“不要骚扰对方”这类否定性安全说明被误拒绝。
5. 对跟踪、公司楼下等待、连续骚扰、灌醉、伪装偶遇、偷拍等危险建议新增了服务端规则和回归用例。

### 知识库边界

- 聊天回复仅注入 `SAFE_SOURCE_PATHS` 中的四份人工审核安全卡：`cold-recovery.md`、`date-invitation.md`、`reply-rewrite.md`、`reply-suggestion.md`。
- 研究 CLI 会逐份分析 `knowledge/` 下的全部 Markdown 文档。
- 因此当前实现应准确表述为“模型意图 + 安全白名单词法检索 + 同义扩展”，不是全知识库可直接回复的 Agent RAG。

## 4. 会话与数据存储

`thread_store.py` 新增业务会话元数据表：

- `thread_id` 为主键，`owner_id`、`title`、创建/更新时间为业务字段。
- 创建会话采用插入后核对 owner 的方式，避免两个设备使用同一新 `thread_id` 时互相覆盖归属。
- 为 `owner_id, updated_at` 新增索引，列表查询不再扫描所有 LangGraph checkpoints。
- 本地无 `DATABASE_URL` 时使用 `data/threads.sqlite`；Compose 生产路径通过 PostgreSQL 创建同名表。

LangGraph checkpoint 仍用于保存工作流状态与消息窗口，不再承担会话查询索引。

## 5. 研究 CLI

命令入口：

```powershell
cd backend
py -m app.research --output ..\data-local\research-report.json
```

功能：

- 每个 Markdown 独立调用模型；一份文档失败会写入 `failed` 状态，其他文档继续。
- 默认生成 `OUTPUT.checkpoint.json`，相同路径且 SHA-256 不变的成功记录会直接复用；源文件变更后自动重跑。
- `--estimate` 不调用模型，只输出文档数、请求数和 token 粗估。
- 报告不保存原文；若模型回显连续 24 个以上原文字符，则只记录 `output_rejected`。

当前全量估算结果：51 份文档、51 次模型请求、约 31,812 输入 token、15,300 输出 token。

## 6. 前端与接口协议

- 前端开发服务地址由 `http://127.0.0.1:8000` 改为 `http://127.0.0.1:8001`。
- 回复标签由 `result.skill` 改为 `result.intent`。
- 聊天接口仍使用 SSE，但协议现为 `start -> complete -> end`；不再发送伪造的 `token` 事件。

这意味着当前 SSE 是“完成事件通知”，并非逐 token 真流式。若产品需要逐 token 体验，需要在安全审核与真实模型流式输出之间另行设计协议。

## 7. 部署与模型配置

- 增加 `MODEL_PROVIDER=deepseek|longxia`。
- DeepSeek 和 Longxia 各自拥有 `BASE_URL`、`API_KEY`、`MODEL` 配置项；默认模型为 `deepseek-v4-flash`，Longxia 可选 `gpt-5.6-terra`。
- Docker Compose 仍包含 PostgreSQL，后端通过 `DATABASE_URL` 使用 PostgreSQL checkpoint 与业务线程表。

## 9. 需要人工确认的交付项

以下内容在当前提交中存在，应在推送、部署或后续整理前明确处理：

1. `deploy/.env` 仍被 Git 跟踪。该文件的真实值未写入本文，但环境文件不应作为公开仓库交付物；已暴露的模型 Key 应先轮换，再停止跟踪并决定是否清理历史。
2. `data-local-debug/checkpoints.sqlite` 被提交为二进制调试数据，不属于运行时代码或部署必需文件。
3. `knowledge/pua-knowledge-sharing` 被提交为 Gitlink（submodule 记录），但仓库未包含对应 `.gitmodules` 映射。普通 clone 后无法可靠初始化该知识目录；需要决定将其作为正式子模块、普通目录版本化，或通过部署时只读挂载提供。
4. 当前设备 UUID 仍不是完整的公网身份认证方案；公网部署前仍需补充 HTTPS、认证/签名令牌、限流与模型调用配额。
